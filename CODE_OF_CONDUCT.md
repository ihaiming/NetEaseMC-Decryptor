# 贡献交流守则

感谢你关注本项目！这是一个个人独立维护的轻量工具。为了大家能愉快、高效地协作，请花一分钟了解这里的交流原则。

## 基本原则：像和同事面对面交流一样

-   **友善与尊重**：请使用包容、专业的语言。我们欢迎任何经验水平的贡献者。
-   **对事不对人**：对代码、方案有异议时，请聚焦于技术本身，不要评价他人的水平、身份或动机。
-   **耐心倾听**：在提出你的想法前，先试着理解已有的设计决策和项目约束。

## 对于 Issue 和 Pull Request 的建议

这是我作为维护者唯一能进行管理的区域。为了确保你的想法能被高效处理，请：

**提交 Issue 时：**
-   清晰描述你遇到的具体问题，或你期望看到的功能。
-   如果是 bug，请提供尽可能详细的复现步骤和环境信息（浏览器版本、操作系统等）。

**提交 Pull Request 时：**
-   **先讨论，再动手**：任何重大的改动（如引入依赖、重构架构、大幅变更 UI），请务必先开 Issue 沟通。这能避免你耗费心力做出的改动，却因为不符合项目方向而无法合并。
-   **保持项目风格**：本项目坚持**零依赖、单文件、原生 JavaScript**。引入任何新的外部库都很难被接受。
-   **明确你的改动**：在 PR 描述中，说清楚你做了什么，以及为什么这么做。

## 不可接受的行为

以下行为将可能导致你的 Issue 或 PR 被直接关闭或删除，恕不另行通知：

-   发布垃圾信息、广告或无意义的灌水内容。
-   使用侮辱性、歧视性或性暗示的语言。
-   发布任何他人的私人信息。
-   在贡献中尝试植入恶意代码或后门。

## 维护者的职责

作为维护者，我的目标是让这个项目保持健康。我会：
-   尽力阅读和回复每一个 Issue 和 PR。
-   对于不符合项目定位或本守则的贡献，我会明确说明理由并关闭它们。
-   对我无法处理的权限问题（如 GitHub 账号级别的问题），我无能为力。

如果你遇到了让你感到不适的情况，请通过 Issue 直接向我反映。我会在力所能及的范围内处理。

**感谢你的理解和贡献，让这个项目变得更好。**
